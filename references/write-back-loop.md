# 活的记忆闭环（Write-Back Loop）

DESIGN.md 最容易被浪费的用法，是当成一次性下载的死文件。真正的价值在于把它当产品的视觉记忆，随项目迭代逐步变厚。

一句话：**生成 → 对照 Do-Don't 自审 → 把新组件规则回写 DESIGN.md。**

## 四步闭环

**第一步 生成。** 选一套设定集，`heige-design build <slug>` 拿提示词，让 agent 读 DESIGN.md + exports/variables.css，先复述 5 条视觉原则再写页面。

**第二步 截图自审。** 页面做完截图，让 agent 对照该套的 `## Do's and Don'ts` 逐条检查：
- 是否用了规定的主色、字体、圆角、间距。
- 是否出现了 DESIGN.md 明确禁止的模式（紫蓝渐变、居中卡片三件套、日文字体渲染中文）。
- 哪些组件偏离了品牌气质。
- 有没有复现该套写死的签名时刻。

**第三步 回写。** 自审中沉淀出的新组件规则（例如你为这个项目新做了一个定价卡的处理方式），补进 DESIGN.md 的 `components:` 和 `## Components` 章节，作为该套在你项目里的新记忆。

**第四步 收口校验。** `heige-design lint <slug>` 确认回写后仍 0 error（token 引用没断、对比度没破）。

## 回写时的纪律

- 新组件按 `{colors.x}` / `{typography.x}` / `{rounded.x}` 引用既有 token，别硬编码颜色。
- 一次只加一个组件条目，别把状态嵌套进对象（`-active` / `-hover` 单列条目）。
- 回写前后各跑一次 lint，对比 `heige-design diff` 看清楚改了什么。
- 数字/金额单元格保持等宽 tabular；中文字体栈保持系统兜底。

## 为什么值得

设计师脑子里的"为什么这个按钮不能太圆""为什么主色只给一个动作"这些决策，如果只停在脑子里，AI 永远学不到。写进 DESIGN.md 之后，它就变成 agent 能持续遵守的产品人格。项目越做越久，这份设定集越厚，AI 产出越稳。
